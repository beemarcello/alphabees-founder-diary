How we built a design system that works across five brands without going insane

One Figma variable library. Five brand identities. Zero duplication. Here's the exact setup we use and the mistakes we made along the way.

This week was all about design systems and one question we kept asking ourselves: how do you maintain visual consistency across multiple brands without maintaining five separate systems?

The trigger was simple. We had just launched our third brand under the Alphabees umbrella and realised that we were already duplicating work. Colors here, type scales there, the same components rebuilt three times.

The problem with separate design files

When you have more than one brand, the natural instinct is to keep everything separate. A Figma file per brand, a component library per brand. Clean. Isolated. Easy to hand off.

The problem shows up six months later when you update a button radius in one brand and then spend three days updating it everywhere else. We've been there.

"The best design systems are the ones you forget are there. They just work, everywhere, all the time."

What we tried first

  • Shared component library with brand-specific overrides per file
  • A single "master" file that each brand file linked to
  • Tokens plugins like Tokens Studio to sync variables across files

The setup that actually works

After several failed attempts, here's the architecture we landed on. It uses Figma's native variable system and one foundation library that every brand file subscribes to.

1
Foundation Library — brand-agnostic primitives

Base spacing scale, font sizes, border radii. Nothing brand-specific. All other libraries subscribe to this one.

2
Brand Token Layer — one file per brand

Each brand overrides the foundation with its own colors, typeface and logo. Still subscribes to the same component structure.

3
Component Library — built once, themed per brand

Buttons, cards, forms — all built with variable references so they automatically shift when the brand token changes.

4
Page files — one per project

Consumes the component library and brand tokens. Never touches the foundation directly.

The rule we follow: if you find yourself editing the same thing in two places, it belongs in the foundation library — not in the brand files.

What we learned

Key Takeaways
1
Start with the foundation layer first — even if you only have one brand. You will have more later.
2
Variables in Figma are only as powerful as your naming convention. Invest one afternoon in naming rules before you build anything.
3
Avoid plugin dependencies for the core system. Plugins break. Native Figma features don't (usually).
What's next

Frequently Asked
Do you need to be on Figma's paid plan to use variables? +
Variables are available on all Figma plans including the free tier. The limitation on free plans is the number of variable collections you can create, but for most small teams it's more than enough to get started.
How do you handle dark mode across brands? +
Each brand token file has a light and dark mode variable set. Components reference semantic tokens like color/surface/primary instead of raw hex values. Switching modes swaps the entire token set in one click.
What happens when a brand needs a component that others don't? +
Brand-specific components live in the brand's own component file, not in the shared library. They still consume the same foundation tokens, so they stay visually consistent. We only add to the shared library when at least two brands need the same component.
